1. 사전 준비 (한 번만)

1.1 Git 설치 & 문서 다운로드

  1. https://git-scm.com/download/win 접속 → 다운로드 → 기본값(Next)으로 설치
  2. 바탕화면 우클릭 → "Git Bash Here"
  3. 아래 2줄을 차례로 붙여넣고 Enter: 
    cd /c/
git clone https://github.com/Achil7/purchaseWeb.git campmanager-manual
  4. 완료되면 C:\campmanager-manual\docs\에 이 문서가 생깁니다.

1.2 장애 시 최신 문서로 업데이트

cd /c/campmanager-manual
git pull

1.3 VSCode + AI 확장 설치 (한 번만)

AI가 에러 이미지를 받는 것에 그치지 않고 실제 프로젝트 코드를 직접 보면서 답할 수 있어야 정확한 조치가 나옵니다. 따라서 코드 편집기(VSCode)에 AI 확장을 설치해 사용합니다.

(1) VSCode 설치

  1. https://code.visualstudio.com 접속 → "Download for Windows" 클릭 → 기본값으로 설치
  2. 설치 완료 후 VSCode 실행
  3. 상단 메뉴 File → Open Folder → C:\campmanager-manual 폴더 선택 → 열기

(2) AI 확장 설치 (아래 2개 중 최소 1개)

VSCode 왼쪽 사이드바의 네모 4개 모양(Extensions) 아이콘 클릭 → 검색창에 이름 입력 → Install 버튼.

확장 이름검색어필요한 구독
Claude Code (1순위 추천)Claude CodeClaude Pro/Max (claude.ai)
Codex (대안)CodexChatGPT Plus (chatgpt.com)
Claude Code 권장: 코드 분석·프로젝트 탐색·터미널 실행까지 한 번에 되며, 이미지 붙여넣기로 에러 화면을 바로 보여줄 수 있습니다. Codex는 대안으로 둡니다.

(3) 로그인

확장 설치 후 VSCode 좌측 사이드바에 해당 확장 아이콘이 생깁니다. 클릭하면 로그인 버튼이 보입니다.

  • Claude Code → "Sign in with Claude" → 브라우저로 이동해 claude.ai 계정 로그인 → VSCode로 돌아오면 자동 연결
  • Codex → "Sign in" → 브라우저로 이동해 ChatGPT 계정 로그인 → VSCode로 돌아오면 자동 연결

(4) 동작 확인 (테스트)

  1. VSCode에서 AI 확장 패널 열기
  2. 채팅창에 안녕 입력 → 답이 오면 정상
  3. 아무 이미지나 채팅창에 Ctrl+V로 붙여넣기 → 이미지가 보이면 OK

2. 서버 접속 방법

서버 명령은 브라우저의 AWS 콘솔 웹 터미널에서 실행합니다. 별도 프로그램이나 키 파일은 필요 없습니다.

2.1 접속 절차

  1. https://aws.amazon.com 접속 → 우측 상단 "콘솔에 로그인"
  2. 상단 검색창에 EC2 입력 → EC2 클릭
  3. 우측 상단 리전이 서울(ap-northeast-2)인지 확인
  4. 왼쪽 메뉴 "인스턴스(실행 중)" → CampManager 서버 체크박스 클릭
  5. 상단 주황색 "연결" 버튼 클릭
  6. "EC2 Instance Connect" 탭 선택 → 아래쪽 "연결" 버튼 클릭
    "SSH 클라이언트" 탭이 아닙니다.
  7. 검은 터미널 창이 열립니다. 아래와 비슷하면 성공: 
    ubuntu@ip-172-31-XX-XX:~$

3. 증상별 진단표

브라우저 화면에서 가장 가까운 증상을 찾아 오른쪽 시나리오로 이동합니다. 모든 증상은 브라우저에서 직접 보이는 것을 기준으로 했습니다.

브라우저 화면에서 이런 것이 보인다면이동
주소를 쳐도 아예 안 열림 / "이 페이지에 연결할 수 없음" / 무한 로딩시나리오 4.1
"502 Bad Gateway" / "503 Service Unavailable" / "504 Gateway Timeout" 글자시나리오 4.2
"404 Not Found" / "페이지를 찾을 수 없음"시나리오 4.3
주소창 자물쇠가 빨간색 / "이 연결은 비공개가 아닙니다" / NET::ERR_CERT시나리오 4.4
로그인 버튼을 눌러도 "아이디/비밀번호" 류 오류 메시지시나리오 4.5
로그인은 성공인데 대시보드에 머물지 못하고 바로 로그아웃됨 / 자꾸 튕김시나리오 4.6
로그인은 되는데 목록이 비어있음 / 빨간 에러 팝업 / "데이터 불러오기 실패"시나리오 4.7
리뷰 이미지 업로드가 실패 / "업로드 실패" / "파일이 너무 큽니다"시나리오 4.8
사이트가 극도로 느림 / 시트 버벅 / 가끔 안 뜨다가 다시 뜨다가 반복시나리오 4.9
데이터가 통째로 사라졌다 / 어제까지 있던 것들이 싹 없어짐시나리오 4.10 긴급
위 표에 없는 증상이면 [6. AI에 질문하는 법]으로 이동해 AI에 먼저 물어봅니다. AI 답변으로도 해결 안 되면 개발자에게 연락합니다.

4. 시나리오별 대응

모든 시나리오는 동일한 순서를 따릅니다:

  1. ① 증상 — 브라우저에서 보이는 것
  2. ② 캡처 — 나중에 AI에 보낼 자료 미리 찍기
  3. ③ 서버 상태 체크 — 터미널에서 명령 실행
  4. ④ 결과 분기 — 체크 결과에 따라 다음 단계 선택
  5. ⑤ 대응(해결)
  6. ⑥ 안 풀리면 AI에 질문
  7. ⑦ 복구 확인

4.1 사이트가 아예 안 열림

① 증상

이 페이지에 연결할 수 없습니다
또는: ERR_CONNECTION_TIMED_OUT
또는: ERR_CONNECTION_REFUSED
또는: 하얀 화면에서 로딩만 계속

② 캡처

  1. 브라우저 에러 화면 (주소창 포함)
  2. 같은 PC에서 다른 사이트(예: naver.com)는 열리는지 확인 후 그 화면
    • 다른 사이트도 안 열리면 → 내 PC 인터넷 문제. 공유기 재시작부터.

③ 서버 상태 체크 (터미널에서 한 줄씩)

docker ps

예상 결과: campmanager-app이 목록에 있고 STATUS가 Up ...이면 정상.

sudo systemctl status nginx

예상 결과: 초록색 active (running)이 보이면 정상.

curl http://localhost:5000/health

예상 결과: {"status":"OK"}이면 정상.

④ 결과 분기

[docker ps 에서 campmanager-app 이] 목록에 없음 또는 Exited → ⑤-A Restarting → 시나리오 4.9 [systemctl status nginx 가] active (running) 이 아님 → ⑤-B [위 둘 다 정상인데 사이트 안 열림] → ⑤-C

⑤ 대응

A. 서비스 기동

cd ~
docker compose up -d

예상 결과: Started 메시지, 10초 뒤 사이트 복구.

B. 웹서버 재시작

sudo systemctl restart nginx

C. 일반 재시도

docker restart campmanager-app

20초 대기 후 브라우저 새로고침.

⑥ 안 풀리면 AI에 질문

docs/LLM_EMERGENCY_CONTEXT.md 를 먼저 읽어주세요.
사이트가 아예 안 열립니다. ②에서 찍은 캡처와 ③의 명령 실행 결과를 올립니다.
원인과 실행할 명령어를 알려주세요.

⑦ 복구 확인

  • 브라우저에서 서비스 주소 접속 시 정상 로딩
  • 로그인 성공

4.2 502 / 503 / 504 에러

① 증상

502 Bad Gateway
또는: 503 Service Unavailable
또는: 504 Gateway Timeout

② 캡처

  1. 브라우저의 에러 화면 (글자·코드 번호 잘 보이게)
  2. 몇 번 새로고침해 봤을 때 간헐적으로 뜨는지 / 항상 뜨는지 기록

③ 서버 상태 체크

docker ps
curl http://localhost:5000/health
docker logs --tail 100 campmanager-app

④ 결과 분기

[docker ps 의 STATUS 가] Up → ⑤-A (웹서버 재시작) Exited / 목록에 없음 → ⑤-B (서비스 기동) Restarting → 시나리오 4.9 [curl 결과가] {"status":"OK"} → ⑤-A Connection refused / 오래 걸림 → ⑤-B

⑤ 대응

A. 웹서버 재시작

sudo systemctl restart nginx

B. 서비스 재기동

cd ~
docker compose up -d

20초 대기 후 브라우저 새로고침.

C. 둘 다 해도 안 되면 완전 재시작

docker restart campmanager-app
sudo systemctl restart nginx

⑥ 안 풀리면 AI에 질문

docs/LLM_EMERGENCY_CONTEXT.md 를 먼저 읽어주세요.
브라우저에 502(또는 503/504) 에러가 뜹니다. 캡처와 docker ps, docker logs 결과를 올립니다.
원인과 실행할 명령어를 알려주세요.

⑦ 복구 확인

  • 브라우저 재접속 시 502/503/504 사라짐
  • 로그인 성공 후 대시보드 이동

4.3 404 Not Found / 페이지를 찾을 수 없음

① 증상

404 Not Found
또는: 페이지를 찾을 수 없습니다
또는: Cannot GET /xxx

② 캡처

  1. 404가 뜬 브라우저 화면 (주소창의 URL 전체가 보이게)
  2. 같은 도메인의 다른 페이지(예: 로그인 페이지)는 열리는지 확인 후 그 화면

③ 범위 확인 (가장 먼저)

  • 특정 페이지 1개만 404 → 사용자가 잘못된 링크를 클릭했을 가능성 큼. 메인에서 다시 진입 유도.
  • 모든 페이지가 404 → 서버 문제. 아래 체크로 진행.

④ 서버 상태 체크

docker ps
curl http://localhost:5000/health

⑤ 결과 분기

[curl 결과가] {"status":"OK"} → ⑥-A (방금 배포된 코드 문제) 응답 없음/다른 응답 → ⑥-B (재시작)

⑥ 대응

A. 배포 직후라면 롤백 필요 — 개발자 연락.

B. 일반 재시도

docker restart campmanager-app

20초 대기 후 해당 페이지 재접속.

⑦ 안 풀리면 AI에 질문

docs/LLM_EMERGENCY_CONTEXT.md 를 먼저 읽어주세요.
특정 페이지(또는 전체)가 404로 뜹니다. URL이 보이는 캡처와 docker ps, curl 결과를 올립니다.
최근에 배포를 한 적이 있는지도 함께 확인해주세요.

⑧ 복구 확인

  • 해당 페이지 정상 로딩
  • 다른 페이지들도 정상

4.4 보안 경고 / 인증서 오류

① 증상

이 연결은 비공개 연결이 아닙니다
또는: NET::ERR_CERT_DATE_INVALID
또는: 주소창 자물쇠가 빨간색/X자

② 캡처

  1. 브라우저 경고 화면 ("고급" 버튼 누르지 말고 그대로)
  2. 주소창 자물쇠 아이콘 클릭 → "인증서가 유효하지 않습니다" 상세 화면

③ 서버에서 만료일 확인

sudo certbot certificates

예상 결과: Expiry Date 줄에 날짜가 보임. 오늘 날짜보다 과거면 만료.

④ 대응 — 인증서 갱신

sudo certbot renew
sudo systemctl restart nginx

예상 결과: Successfully renewed 메시지. 브라우저 재접속 시 자물쇠가 초록색.

⑤ 갱신 실패 시 AI에 질문

docs/LLM_EMERGENCY_CONTEXT.md 를 먼저 읽어주세요.
certbot renew 가 실패했습니다. 터미널 출력 전체를 캡처해서 올립니다.
파일 편집 없이 가능한 조치만 알려주세요.

AI도 "개발자 연락"이라고 답하면 → 개발자 연락.

⑥ 복구 확인

  • 주소창 자물쇠가 초록/회색(정상)
  • sudo certbot certificates 만료일이 약 90일 뒤로 갱신됨

4.5 로그인이 안 됨

① 증상

아이디 또는 비밀번호가 올바르지 않습니다
또는: 인증 처리 중 오류가 발생했습니다
또는: 로그인 버튼을 눌러도 아무 반응 없음

② 캡처

  1. 로그인 실패 시 브라우저 화면 (에러 문구 포함)
  2. 브라우저에서 F12 → "Network" 탭 열어둔 상태로 로그인 재시도 → 빨간 줄 요청 클릭 → 오른쪽 "Response" 탭 캡처

③ 범위 확인

  • 나만 안 됨 (다른 직원은 됨) → ⑤-A
  • 전 직원 안 됨 → ⑤-B

④ 서버 상태 체크 (전원 안 될 때만)

curl http://localhost:5000/health
docker logs --tail 100 campmanager-app

⑤ 대응

A. 나만 안 될 때

4.7 데이터가 안 보이거나 에러 팝업이 뜸

로그인은 되는데 화면에 문제가 있는 모든 경우를 여기서 다룹니다.

  • 페이지에 "undefined" / "null" 같은 이상한 문자
  • 배포 직후부터 이상 증상이 시작됨

② 캡처

  1. 이상 현상이 보이는 화면
  2. 브라우저 F12 → "Console" 탭: 빨간 글자 전부 (스크롤하며 여러 장 캡처)
  3. 브라우저 F12 → "Network" 탭: 빨간 줄(실패 요청) 클릭 → "Response" 탭 캡처

③ 범위 확인

  • 전 사용자가 똑같은 이상 → ⑤-A
  • 특정 사용자만 이상 (다른 사용자는 정상) → ⑤-B
  • 배포 직후부터 → ⑤-C

④ 서버 상태 체크 (⑤-A 또는 ⑤-C일 때)

curl http://localhost:5000/health
docker logs --tail 150 campmanager-app

⑤ 대응

A. 전원 이상 — 일반 재시도

B. 특정 사용자만 이상

  2. 해당 사용자가 캠페인/품목에 정상 배정되어 있는지 확인
  3. 사용자에게 브라우저 Local Storage Clear + 재로그인 요청

C. 배포 직후부터 — 마이그레이션 시도

docker compose exec app sh -c "cd /app/backend && npx sequelize-cli db:migrate"

4.8 이미지 업로드 실패

① 증상

  • "파일 크기가 10MB를 초과했습니다"
  • "413 Request Entity Too Large" 화면
  • 업로드 버튼 눌러도 아무 반응 없음

② 캡처

  1. 업로드 실패 시 화면
  2. 업로드 시도한 파일의 크기 (파일 탐색기에서 파일 우클릭 → 속성)
  3. 브라우저 F12 → "Console" 탭 빨간 글자

③ 범위 확인 (먼저 해볼 것)

  • 파일이 10MB를 넘는가? → 넘으면 크기를 줄여 재업로드. 해결.
  • 파일 형식이 jpg/png/gif/webp가 맞는가? → 아니면 다른 파일로 재시도.
  • 한 번에 50개를 넘지 않았는가? → 나눠서 업로드.
  • 위 3개가 전부 OK인데 실패하면 → ⑤로 진행.

④ 서버 상태 체크

4.10 데이터 대량 삭제·손상 의심 (긴급)

이 시나리오는 "아무것도 하지 않는 것"이 가장 중요합니다. 섣부른 재시작·재배포가 상황을 돌이킬 수 없게 만들 수 있습니다.

① 증상

  • 캠페인/구매자 목록이 통째로 비어 있음
  • 특정 테이블의 모든 데이터가 사라짐
  • 어제까지 있던 이미지가 대량으로 사라짐
  • 직전에 금지 명령을 실수 실행한 기억이 있음
  • 사용자 계정이 모르는 사이에 생성됨 / 해킹 의심

② 즉시 순서 (반드시 이 순서)

  1. 재시작·재배포 금지. docker restart, docker compose, pull 모두 중단.
  2. 현재 화면을 최대한 많이 캡처해 둠 (브라우저, 터미널 모두).
  3. 개발자에게 즉시 전화. 문자 아닌 전화. 응답 없으면 카톡/문자로 "긴급: 데이터 손상 의심".
  4. AWS 콘솔 → RDS → "스냅샷" 메뉴 → 최근 자동 스냅샷 목록 캡처만 (삭제/복원 버튼 누르지 않음).
  5. AWS 콘솔 → S3 → 버킷 → "속성" 탭에서 "버전 관리"가 켜져 있는지 확인 후 캡처.

③ 절대 하지 말 것

  • 서비스 재시작/재배포
  • RDS 재시작/스냅샷 삭제/인스턴스 종료
  • S3 버킷 설정 변경
  • AI 답변대로 복구 명령 실행 (복구는 개발자만)
  • 이상해 보이는 데이터 수정/삭제

④ AI에는 진단 질문만

docs/LLM_EMERGENCY_CONTEXT.md 를 먼저 읽어주세요.
데이터 손상이 의심됩니다. 현재 화면 캡처를 올립니다.
어떤 명령도 실행하지 않을 것이며 개발자에게 이미 연락했습니다.
"지금 하면 안 되는 행동"만 알려주세요. 복구 명령은 절대 제시하지 말아주세요.

⑤ 이후

복구는 개발자가 RDS 스냅샷 또는 Point-in-Time Recovery로 수행합니다. 복구 완료 후 표본 데이터 1~2건만 직접 확인합니다.

6. AI(LLM)에 질문하는 법

명령어만으로 안 풀리는 장애는 AI에 에러 이미지를 보내 진단받습니다. VSCode 확장을 쓰면 AI가 프로젝트 코드를 직접 보면서 답하기 때문에 훨씬 정확합니다.

6.1 준비 확인

섹션 1.3에서 아래가 모두 끝나 있어야 합니다.

  • VSCode 설치 + C:\campmanager-manual 폴더 열기
  • Claude Code 또는 Codex 확장 설치 + 로그인 완료
  • 테스트 채팅 동작 확인

6.2 스크린샷 찍기

OS단축키결과
WindowsWin + Shift + S영역 드래그 → 클립보드 복사 → AI 채팅창에서 Ctrl+V
MacCmd + Shift + 4영역 드래그 → 바탕화면 저장 → AI 창에 드래그

터미널 캡처 시 브라우저 F11(전체화면) 후 글자가 잘리지 않게 찍습니다.

6.3 질문 표준 절차 (VSCode에서)

  1. VSCode 열기 → File → Open Folder → C:\campmanager-manual 열려있는지 확인
  2. 왼쪽 사이드바의 Claude Code(또는 Codex) 아이콘 클릭 → 채팅 패널 열기
  3. 채팅창에 Ctrl+V로 에러 스크린샷 1~3장 붙여넣기
  4. 아래 6.4 고정 템플릿을 복사해서 채팅창에 붙여넣기
  5. Enter → 답변 대기 (AI가 자동으로 프로젝트 코드를 탐색해 문맥 파악)
  6. 답변의 6.5 안전 체크 확인 후 명령어 실행
  7. 터미널에서 명령 실행한 결과를 다시 스크린샷 → 같은 채팅창에 붙여넣고 다음 단계 받기

6.5 AI 답변 안전 체크 (실행 전 필독)

아래 단어가 답변에 하나라도 보이면 즉시 중단하고 개발자 연락:
  • rm -rf, rm -r
  • --volumes (docker system prune과 함께)
  • docker volume rm / prune
  • DROP, TRUNCATE, DELETE FROM
  • sudo reboot, sudo shutdown, sudo poweroff
  • vi, vim, nano로 파일 편집
  • sed -i, awk로 설정 파일 수정
  • chmod 777
  • git push, docker push, docker build
아래 경우도 일단 중단:
  • 답변이 "파일을 여세요"로 시작
  • 같은 조치 3번 반복해도 동일 증상
  • 명령 의미를 전혀 이해할 수 없음 → AI에 "이게 정확히 뭘 하는 명령인지 비전공자 기준으로 설명해주세요" 추가 질문
  • AI가 직접 파일을 수정하겠다고 제안할 때 — Claude Code는 실제로 파일 편집을 시도할 수 있으므로 "코드 수정 없이 서버 명령어만 알려주세요"로 다시 요청
아래 명령만 있으면 안전하게 실행:
  • docker ps, docker logs, docker stats, docker inspect
  • docker restart, docker compose up -d, docker compose pull
  • curl, df -h, free -h
  • systemctl status|restart nginx
  • certbot certificates|renew
  • docker image prune -a -f, docker container prune -f

6.6 대화 이어가기

명령 실행 결과 캡처를 같은 채팅창에 붙여넣고 "결과입니다. 다음 단계를 알려주세요." 멘트 추가. 새 대화를 시작하면 AI가 앞 흐름을 잊으니 반드시 이어서 진행합니다.

7. 절대 하지 말 것

파일·데이터 삭제 계열

  • rm -rf /, rm -rf *
  • docker system prune -a --volumes — 저장 데이터 영구 삭제
  • docker volume rm, docker volume prune
  • DROP TABLE, DROP DATABASE, TRUNCATE, DELETE FROM

서버 중지 계열

  • sudo reboot, sudo shutdown, sudo poweroff
  • AWS 콘솔 "인스턴스 종료(Terminate)" — 복구 불가
  • AWS 콘솔 RDS 인스턴스/스냅샷 삭제
  • AWS 콘솔 S3 버킷 삭제 또는 "버킷 비우기"

권한·설정 변경

  • AWS IAM에서 사용자/권한 삭제
  • AWS 보안 그룹 규칙 삭제
  • 도메인(Route 53) 설정 변경
  • chmod 777

반복 재시도 금지

  • 같은 조치 3회 이상 반복 — 원인이 다른 데 있을 가능성
  • "혹시 몰라" 수 초 간격 재시작 — 데이터 꼬일 수 있음
  • AI 답변을 안전 체크 없이 복사·붙여넣기

비밀 정보 노출

  • .env 파일 내용을 카톡/이메일로 전송하지 않음
  • AWS 키, DB 비밀번호가 포함된 캡처를 AI에 업로드하지 않음
    • docker exec campmanager-app env 결과에는 값이 포함되므로 AI 업로드 금지

8. 개발자에게 연락할 상황

아래 중 하나라도 해당하면 즉시 연락합니다. 복잡한 상황은 AI가 아닌 개발자가 처리하는 것이 빠르고 안전합니다.

8.1 즉시(전화) 연락

  • 데이터 손상/유실 의심 (시나리오 4.10)
  • 섹션 7의 금지 명령을 실수로 실행한 경우
  • AWS 콘솔 로그인 불가 (계정 잠김, MFA 분실)
  • RDS/EC2/S3가 AWS 측에서 장애 상태로 표시됨
  • 해킹 의심 징후 (비정상 트래픽, 의문의 계정 생성)

8.2 문자/카톡 (답변 대기 가능)

  • AI가 "개발자 연락"이라고 답한 경우 (AI 답변 캡처 동봉)
  • 같은 조치 3회 반복 후에도 복구 안 됨
  • 파일 편집/코드 수정이 필요한 해결책만 보임 (Nginx 설정, 환경변수 등)
  • 마이그레이션 실패

8.3 전달하면 빠른 정보

  • 증상 요약: "X시 XX분부터 로그인이 전혀 안 됩니다"
  • 지금까지 실행한 조치: "docker restart 2번, pull 1번 했습니다"
  • 브라우저 에러 스크린샷
  • docker ps / docker logs --tail 200 스크린샷
  • AI와 대화했으면 그 답변 스크린샷
